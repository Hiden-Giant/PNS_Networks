# PNS Networks 포워딩 영업 및 운임 관리 플랫폼 구축 계획서

- 문서 버전: 1.0
- 작성 기준일: 2026-07-17
- 기준 문서: `PNS Networks 포워딩 영업 및 운임 관리 플랫폼 제안서_V2.0_2026.07.14.pdf`
- 적용 프로젝트: `cargo_project_lionking`

## 1. 계획 목적

현재 Next.js 기반 UI 목업을 실제 포워딩 영업 및 운임 관리 업무에 사용할 수 있는 플랫폼으로 전환한다. 라우트, 스케줄, 운임, 고객, 견적 데이터를 통합 자산화하고, 영업 담당자가 최신 운임과 스케줄을 조회해 견적을 생성·승인·발송하며 그 이력을 관리할 수 있도록 한다.

플랫폼은 우선 핵심 업무와 데이터 기반을 구축하고, 이후 외부 운임·스케줄 API와 AI 분석 기능을 단계적으로 확장할 수 있는 구조로 설계한다.

## 2. 확정 범위

1. 메인 대시보드
   - 유효·만료·오픈 예정 스케줄 현황

2. 기준 정보 관리
   - 지역, 국가, 항구, 공항 코드

   - 부가 서비스 및 비용 코드
   - Incoterms, 통화, 운송 방식, 과금 단위
   - 라우트, 스케줄, 운임, 마진 규칙
3. 고객 및 사용자 관리
   - 고객사와 담당자
   - 내부 사용자, 역할, 권한
4. 스케줄 및 요율 조회
   - 출발지, 도착지, 운송 방식, 날짜, 화물 조건 검색
   - 판매 가능 운임과 부가 서비스 표시
5. 견적 관리
   - 견적 요청, 작성, 계산, 승인, 발송, 버전 및 상태 이력
   - 견적 템플릿과 PDF 출력
6. 연동 관리
   - 연동 업체 설정
   - API·수기·엑셀 데이터 수집
   - 동기화 실행 결과와 오류 모니터링
7. 시스템 관리
   - 접근 권한
   - 접속 및 데이터 변경 감사 로그
8. 확장 기반
   - 다국어 적용이 가능한 메시지 구조
   - 향후 AI 검색·비교·분석을 위한 이력 데이터 축적

### 2.2 명시적 제외 범위

제안서 11페이지의 다음 내용은 잘못 기재된 항목으로 보고 구축 범위에서 제외한다.

- 입고 확정
- 창고 로케이션 지정
- 재고 수량 반영
- WMS 재고 관리
- 창고 작업자 및 매장 스태프 역할

외부 고객이 직접 포워딩을 신청하고 전체 배송을 추적하는 고객 포털, 다국어 UI 전체 적용, AI 포워딩 메이트는 후속 단계로 분리한다.

## 3. 구축 원칙

### 3.1 모듈 분리

라우트, 스케줄, 운임을 독립적인 마스터로 관리하고 상품 조회 시 조합한다.

```text
코드북 → 라우트 → 스케줄 → 운임·부대비용 → 판매 조건 → 견적
```

동일한 지역·선사·경로 정보를 반복 입력하지 않으며, 각 마스터의 변경 이력과 유효기간을 관리한다.

### 3.2 원가와 판매가 분리

매입 원가, 부대비용, 마진·할인 규칙, 고객 판매가를 구분한다. 견적 확정 시 계산 근거를 스냅샷으로 저장해 이후 마스터 데이터가 변경되어도 과거 견적을 재현할 수 있게 한다.

### 3.3 출처와 이력 보존

수기, 엑셀, 외부 API 등 데이터 출처를 기록한다. 중요 데이터의 생성·수정·삭제는 사용자, 시각, 변경 전후 값을 감사 로그에 남긴다.

### 3.4 권한 기반 노출

원가와 마진은 권한 있는 사용자만 볼 수 있게 한다. UI 메뉴 숨김뿐 아니라 서버 API에서 권한을 다시 검증한다.

### 3.5 단계적 확장

초기 구축은 단일 Next.js 애플리케이션과 관계형 DB를 사용하는 모듈형 모놀리스로 시작한다. 외부 연동 작업은 비동기 작업 계층으로 분리해 향후 독립 서비스로 확장할 수 있게 한다.

## 4. 사용자와 역할

| 역할 | 주요 권한 |
|---|---|
| 시스템 관리자 | 사용자, 역할, 코드북, 시스템 설정 전체 관리 |
| 운임 관리자 | 라우트, 스케줄, 매입 운임, 부대비용, 마진 규칙 관리 |
| 영업 관리자 | 고객, 견적, 할인 승인, 영업 현황과 마진 조회 |
| 영업 담당자 | 스케줄·판매가 조회, 고객 및 견적 작성·발송 요청 |
| 운영 담당자 | 스케줄 상태, 선사 정보, 연동 오류 확인 및 정정 |
| 감사·조회 사용자 | 허용된 데이터와 감사 로그 읽기 전용 |
| 외부 고객 | 후속 고객 포털 단계에서 별도 정책 적용 |

권한은 `resource + action` 방식으로 세분화한다. 예: `rate.cost.read`, `rate.write`, `quotation.approve`, `audit.read`.

## 6. 데이터 모델

### 6.1 코드 및 위치

| 엔티티 | 핵심 속성 |
|---|---|
| Region | 코드, 이름, 활성 상태 |
| Country | ISO 코드, 지역, 한글·영문명 |
| Location | 국가, 유형(PORT/AIRPORT), UN/LOCODE, IATA, 시간대 |
| Carrier | SCAC/IATA 코드, 유형, 이름, 활성 상태 |
| ServiceCode | 서비스 코드, 이름, 과금 방식, 과세 여부 |
| Incoterm | 코드, 버전, 설명 |
| Currency | ISO 통화 코드, 소수 자릿수 |

### 6.2 운송 마스터

| 엔티티 | 핵심 속성 |
|---|---|
| Route | 출발지, 도착지, 운송 방식, 경유 여부, 상태 |
| Schedule | 라우트, 선사, 선박·편명, ETD, ETA, 판매 기간, 상태 |
| Rate | 라우트·스케줄, 통화, 단위, 매입가, 유효기간, 출처 |
| RateSurcharge | 운임, 비용 코드, 금액·비율, 필수 여부 |
| MarginRule | 적용 대상, 우선순위, 비율·고정액, 유효기간 |
| CustomerRate | 고객, 기준 운임, 특가·할인, 유효기간 |

스케줄 상태는 최소 `DRAFT`, `UPCOMING`, `ON_SALE`, `HOLD`, `EXPIRED`, `CANCELLED`로 관리한다.

| 엔티티 | 핵심 속성 |
|---|---|
| Customer | 법인명, 사업자번호, 국가, 등급, 담당 영업자, 상태 |

| QuotationApproval | 승인자, 결과, 의견, 일시 |
| QuotationDelivery | 수신자, 발송 채널, 발송 결과, 일시 |

견적 상태는 `DRAFT`, `REVIEW_REQUESTED`, `APPROVED`, `REJECTED`, `SENT`, `NEGOTIATING`, `ACCEPTED`, `DECLINED`, `EXPIRED`로 관리한다.

### 6.4 사용자, 연동 및 감사

| 엔티티 | 핵심 속성 |
|---|---|

API 키와 비밀번호는 DB 평문으로 저장하지 않고 배포 환경의 비밀 관리 기능을 사용한다.

## 7. 핵심 업무 흐름

### 7.1 마스터 등록

1. 관리자가 지역·국가·항구·공항·선사 코드를 등록한다.
2. 운임 관리자가 출발지와 도착지를 조합해 라우트를 생성한다.
3. 라우트에 선사, ETD, ETA를 연결해 스케줄을 생성한다.
4. 라우트 또는 스케줄에 매입 운임과 부대비용을 등록한다.
5. 마진 규칙과 고객별 특가를 설정한다.
6. 검증을 통과한 항목만 판매 가능 상태로 전환한다.

### 7.2 스케줄 및 요율 조회

1. 사용자가 출발·도착·운송 방식·날짜·화물 조건을 입력한다.
2. 서버가 유효한 라우트, 스케줄, 운임을 조회한다.
3. 부대비용과 권한별 마진·할인을 계산한다.
4. 영업 담당자에게 판매가와 조건을 표시한다.
5. 선택 결과를 견적 초안으로 전달한다.

### 7.3 견적 처리

1. 고객과 화물 조건을 확정한다.
2. 하나 이상의 스케줄·운임을 견적 라인으로 선택한다.
3. 시스템이 원가, 부대비용, 마진, 판매가를 계산한다.
4. 권한이나 할인 한도에 따라 내부 승인을 요청한다.
5. 승인된 견적을 PDF로 생성하고 고객에게 발송한다.
6. 수정·네고 시 새 버전을 생성한다.
7. 고객 결과와 사유를 기록한다.

### 7.4 데이터 수집

- 수기: 폼 검증 후 즉시 저장
- 엑셀: 업로드 → 열 매핑 → 행별 검증 → 미리보기 → 승인 적용
- API: 수집 → 원본 저장 → 내부 표준 모델 변환 → 중복·오류 검증 → 반영

## 8. 기술 아키텍처

### 8.1 애플리케이션

- Next.js 16 App Router
- React Server Components를 기본으로 사용
- 입력 폼과 즉시 상호작용 영역만 Client Component 사용
- 서버 측 Route Handler 또는 Server Action을 통한 변경 처리
- 모든 변경 요청에 인증, 권한, 입력 검증 적용

구현 전에는 프로젝트의 `node_modules/next/dist/docs`에서 해당 Next.js 16 API 문서를 확인한다.

### 8.2 데이터 및 비동기 처리

- 관계형 데이터베이스: PostgreSQL 권장
- ORM 및 마이그레이션 도구 선정
- 스키마 마이그레이션과 초기 코드 데이터 시드 제공
- 외부 API 수집, 엑셀 대량 반영, 이메일 발송은 비동기 작업으로 처리
- 원본 연동 데이터와 정규화된 업무 데이터를 분리

### 8.3 배포 및 운영

- 개발, 검증, 운영 환경 분리
- CI에서 lint, TypeScript, 단위·통합 테스트, 빌드 실행
- DB 백업과 복구 절차
- 구조화 로그, 오류 추적, 성능 지표, 연동 작업 알림
- 개인정보와 영업 원가 데이터 암호화 및 접근 기록

## 9. 비기능 요구사항

### 9.1 보안

- 세션 기반 인증과 안전한 쿠키 정책
- 서버 측 RBAC 검증
- 원가·마진·고객정보 접근 최소화
- 로그인 실패 제한 및 계정 잠금 정책
- 감사 로그 위변조 방지와 보존 기간 정책
- 업로드 파일 유형·크기·악성 콘텐츠 검사

### 9.2 성능

- 일반 목록과 검색 응답 목표: 정상 부하에서 2초 이내
- 대량 목록 페이지네이션
- 검색 필드 DB 인덱스
- 대시보드 집계 캐시 또는 사전 집계
- 외부 API 지연이 사용자 요청을 직접 차단하지 않도록 비동기화

### 9.3 품질 및 접근성

- 모바일·태블릿·PC 반응형 지원
- 키보드 탐색, 레이블, 포커스, 명도 대비 점검
- 날짜·통화·시간대 표시 규칙 통일
- 모든 업무 상태를 색상 외 텍스트로도 구분
- 한국어를 기본으로 하되 메시지를 코드에서 분리해 다국어 확장 지원

## 10. 단계별 실행 계획

전체 기준 일정은 16주이며, 업무 담당자의 신속한 검토와 외부 연동 사양 제공을 전제로 한다.

### Phase 0. 요구사항 확정 및 기반 정리 - 1~2주

- 실제 사용자·조직·역할 인터뷰
- 운임표, 견적서, 고객 목록, 기존 엑셀 샘플 분석
- 용어, 통화, 단위, 상태, 승인 규칙 확정
- WMS 관련 잘못된 범위 제거 확인
- 화면 목록과 권한 매트릭스 확정
- 개발·검증 환경과 CI 기반 구성

산출물:

- 요구사항 명세서
- 업무 흐름도
- 권한 매트릭스
- 용어·코드북
- 확정 화면 목록

### Phase 1. 데이터·인증·코드북 기반 - 3~5주

- PostgreSQL 및 ORM 구성
- 사용자 인증, 역할, 권한 구현
- 지역, 국가, 위치, 선사, 서비스, Incoterms 코드 관리
- 감사 로그 기반
- 기존 목업 공통 컴포넌트 정리

완료 기준:

- 관리자가 실제 계정을 생성·비활성화할 수 있다.
- 역할별 메뉴와 API 접근이 제어된다.
- 코드북 CRUD와 변경 이력이 동작한다.

### Phase 2. 라우트·스케줄·운임 - 6~9주

- 라우트 CRUD와 중복 검증
- 스케줄 CRUD, 상태, 판매 기간
- 운임, 부대비용, 마진, 고객 특가
- 수기 등록 및 엑셀 업로드
- 현재 `/` 검색 화면을 DB 검색으로 전환
- 대시보드와 운영 알림을 실제 집계로 전환

완료 기준:

- 라우트·스케줄·운임을 독립 등록하고 조합할 수 있다.
- 유효기간과 권한에 맞는 판매가가 검색된다.
- 엑셀 오류가 행 단위로 표시되고 승인된 데이터만 반영된다.

### Phase 3. 고객·견적·영업 흐름 - 10~13주

- 고객사와 담당자 관리
- 견적 요청 및 견적 작성
- 가격 계산과 스냅샷
- 승인·반려·네고·버전 관리
- 견적 템플릿과 PDF
- 이메일 발송 결과와 이력
- 영업·마진 대시보드

완료 기준:

- 검색 결과에서 견적을 생성할 수 있다.
- 견적 계산 근거와 변경 버전을 재현할 수 있다.
- 권한에 따른 승인 후 PDF 발송과 발송 이력이 남는다.

### Phase 4. 외부 연동·안정화·출시 - 14~16주

- 연동 업체 설정과 작업 실행 구조
- 우선순위 선사 또는 데이터 공급자 1개 연동
- 재시도, 오류 원문, 모니터링
- 전체 회귀, 보안, 성능, 접근성 테스트
- 데이터 이관 리허설
- 운영자 교육과 배포·복구 훈련

완료 기준:

- 연동 실패가 업무 데이터에 잘못 반영되지 않는다.
- 운영자가 실패 원인과 재처리 결과를 확인할 수 있다.
- UAT 결함이 합의된 출시 기준 이하이고 복구 절차가 검증된다.

## 11. 테스트 계획

### 11.1 자동화 테스트

- 단위 테스트: 가격 계산, 마진, 할인, 상태 전이, 유효기간
- 통합 테스트: DB 저장, 권한, 엑셀 반영, 외부 API 변환
- E2E 테스트: 로그인 → 검색 → 견적 → 승인 → PDF 발송
- 회귀 테스트: 주요 사용자 역할별 시나리오

### 11.2 업무 검수 시나리오

1. 부산 → 롱비치 해상 라우트와 복수 선사 비교
2. 매입가·부대비용·마진을 포함한 판매가 계산
3. 고객 특가와 일반 운임 우선순위
4. 만료 운임 검색 제외
5. 할인 한도 초과 견적 승인 요청
6. 견적 수정 후 이전 버전 재현
7. 엑셀 중복·통화·날짜 오류 검증
8. 연동 실패 후 재처리와 감사 로그 확인
9. 영업 담당자의 매입 원가 접근 차단

## 12. 데이터 이관 계획

1. 기존 엑셀과 수기 자료 목록화
2. 기준 코드와 중복 식별 규칙 확정
3. 열 매핑 템플릿 제공
4. 샘플 이관과 정합성 보고
5. 전체 리허설
6. 최종 동결 시점 확정
7. 운영 이관과 결과 승인

이관 데이터는 원본 파일, 변환 결과, 오류 보고서를 함께 보존한다. 담당자가 확인하지 않은 추정값은 운영 데이터로 자동 반영하지 않는다.

## 13. 주요 산출물

- 요구사항 및 범위 명세
- 정보 구조와 화면 정의서
- 데이터 모델과 데이터 사전
- API 명세
- 권한 매트릭스
- 소스 코드와 DB 마이그레이션
- 엑셀 업로드 템플릿
- 테스트 계획·결과·UAT 승인서
- 운영·장애·백업·복구 매뉴얼
- 관리자 및 사용자 교육 자료
- 배포 기록과 릴리스 노트

## 14. 의사결정 및 변경 관리

- 제품 책임자 1인이 범위와 우선순위를 최종 승인한다.
- 운임, 영업, 운영, IT 담당자를 업무별 검토자로 지정한다.
- 요구사항 변경은 영향 범위, 일정, 비용, 데이터 변경 여부를 기록한 후 승인한다.
- 제안서와 실제 업무가 충돌하면 실제 업무 확인 결과와 본 계획서의 확정 범위를 우선한다.

## 15. 주요 위험과 대응

| 위험 | 영향 | 대응 |
|---|---|---|
| 기존 엑셀 형식이 부서마다 다름 | 이관 지연·오류 | 표준 템플릿과 사전 프로파일링 |
| 운임 계산 규칙이 담당자 암묵지에 의존 | 판매가 오류 | 사례 기반 규칙 워크숍과 승인 테스트 |
| 원가·마진 권한 누출 | 영업·보안 사고 | 서버 권한 검증, 감사 로그, 최소 권한 |
| 외부 API 사양·품질 불안정 | 일정 지연 | 어댑터 분리, 원본 저장, 수기·엑셀 대안 유지 |
| 범위가 고객 포털·AI까지 조기 확대 | 핵심 구축 지연 | Phase별 출구 기준과 변경 승인 |
| 과거 데이터 품질 불량 | 분석 신뢰도 저하 | 검증·중복 제거·신뢰도 표시 |

## 16. 프로젝트 완료 기준

다음 조건을 모두 충족해야 핵심 플랫폼 구축 완료로 판단한다.

1. 확정된 역할별 인증과 서버 권한 제어가 동작한다.
2. 코드북, 라우트, 스케줄, 운임, 고객을 실제 DB에서 관리한다.
3. 수기·엑셀 데이터 등록과 검증 이력이 남는다.
4. 최신 유효 스케줄과 판매가를 검색할 수 있다.
5. 견적 작성, 승인, PDF 발송, 버전 이력을 처리할 수 있다.
6. 원가, 부대비용, 마진, 판매가 계산을 재현할 수 있다.
7. 주요 변경과 보안 행위가 감사 로그에 기록된다.
8. 우선 외부 연동 1개와 실패 재처리가 검증된다.
9. 핵심 E2E 시나리오와 사용자 검수가 통과한다.
10. 운영 백업·복구와 장애 대응 절차가 확인된다.

## 17. 후속 고도화 로드맵

핵심 플랫폼 안정화와 데이터 품질 검증 후 다음 기능을 별도 승인해 추진한다.

1. 고객 포털과 직접 견적·포워딩 신청
2. 배송·부킹 진행 상태 조회
3. KR·EN·JP 다국어 UI
4. 선사·항공사 연동 확대
5. 고객·라우트별 수주율과 수익성 분석
6. PNS 포워딩 메이트 AI
   - 자연어 스케줄·운임 검색
   - 최저가·최단 기간·최고 마진 비교
   - 과거 단가와 마진 추세 분석
   - 고객 제안 초안 작성

AI 결과는 초기에는 추천과 분석 보조로 제공하고, 가격 확정이나 고객 발송은 담당자의 검토와 승인을 거치게 한다.
